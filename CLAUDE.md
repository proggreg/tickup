## Commands

```bash
pnpm dev          # Start dev server (binds to all interfaces via --host)
pnpm build        # Production build
pnpm lint         # ESLint check
pnpm lint:fix     # ESLint auto-fix

pnpm e2e                        # Run all Playwright e2e tests
pnpm test:api                   # Run only API e2e tests
npx playwright test e2e/todos/new/select-list-test.spec.ts  # Run a single test file
```

E2e tests require a running dev server — Playwright starts one automatically via `webServer` config (`NODE_ENV=test pnpm dev`). Auth state is shared from `user.json` (generated by `e2e/global.setup.ts`, credentials: `testuser@example.com` / `password`).

## Architecture

**Nuxt 4 app** with Vuetify 3, Pinia, and Supabase. Source lives in `app/` (Nuxt 4 convention). The server is Nitro (under `server/`).

### Data flow

- **Backend**: Nitro API routes in `server/api/` talk directly to Supabase via `serverSupabaseClient`. All DB column names are `snake_case`; `ts-case-convert` (`objectToSnake` / `objectToCamel`) handles conversion at the API boundary.
- **Frontend**: One central Pinia store (`app/stores/lists.ts`) owns virtually all state: `lists`, `currentList`, `newTodo`, `todaysTodos`, `overdueTodos`. Components call store actions; stores call `$fetch` against the Nitro API.
- **Auth**: `@nuxtjs/supabase` handles sessions. Protected pages redirect to `/login`; the Supabase callback hits `/confirm`.

### Global types

`index.d.ts` at repo root declares the core interfaces (`Todo`, `List`, `listsState`, `Status`) as globals — no import needed anywhere in the app.

### Key composables

| Composable | Purpose |
|---|---|
| `useDialog()` | `useState` singleton `{ page, open }` — the single dialog open/close mechanism |
| `useNotification()` | `useState` singletons for snackbar: `show`, `message`, `link`. Call `notify(text, { link?, timeout? })` |
| `useShortcutKeys()` | Registers `t` (new todo dialog) and `l` (new list dialog) globally |
| `useAppLayout()` | Returns the layout name based on route + device (default / mobile / todo / login-register) |

### Dialog system

`useDialog()` drives all dialogs. Set `dialog.value = { open: true, page: 'todo' }` to open a dialog. `AppDialog` consumes `page` prop to know when to render. `TodoDialog` and the new-list dialog consume this directly without going through `AppDialog`.

### Keyboard shortcuts

`t` → opens new todo dialog, `l` → opens new list dialog. Handled in `useShortcutKeys`, registered once in `app.vue`.

### New todo flow

`TodoDialog.vue` owns the entire new-todo UI. It uses a local `selectedListId` ref (not `currentList.id`) so the dialog's list picker is independent of the page the user is currently viewing. Before calling `listsStore.addTodo()`, it sets `listsStore.newTodo.listId = selectedListId.value`. The store's `addListId` falls back to `currentList.id` only if `todo.listId` is not already set.

### Styling

Do not use custom CSS classes. Use Vuetify component props, slots, and the `:deep()` scoped override pattern only when a Vuetify prop is insufficient. Prefer Vuetify's built-in spacing (`pa-4`, `ma-2`), colour (`color="primary"`), and layout (`v-row`, `v-col`) utilities over custom class names.

### Vuetify defaults

Global Vuetify component defaults are set in `config/vuetify.ts`. `VBtn` defaults to `variant: 'plain'`, `VTextField` to `variant: 'outlined'`, `density: 'compact'`. Override per-component as needed.

### E2e test conventions

- Tests import `test` and `expect` from `e2e/fixtures/index.ts` (not from `@playwright/test` directly) to get the `listAPI` fixture.
- `listAPI` is a typed wrapper (`e2e/helpers/api/list.ts`) for creating/deleting lists via the API in `beforeEach`.
- Use `getByRole`, `getByTestId`, or `.v-list-item` selectors for Vuetify components — avoid broad `div` locators.
- `data-testid` attributes are on key interactive elements (`new-todo-input`, `list-select`, `create-todo-button`, `dialog-title`, `dialog-close`).
